# 侧边栏新功能效果 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**目标：** 按照《侧边栏新功能效果.md》实现 Codex 风格的“置顶 / 工作空间 / 对话”侧边栏、空白新对话项目选择、项目级批量归档、项目移出侧边栏，以及设置里的“已归档对话”管理页。

**架构：** 不新增独立 Project 后端实体，项目仍然从会话的 `workspaceScope.project_path` 和 WebUI 侧边栏状态推导。新增的“空项目、项目置顶、项目移出侧边栏”都是 WebUI 持久化状态；归档仍然复用现有会话级 `archived_keys`。

**Tech Stack：** React 18、TypeScript、Vite/Vitest、Tailwind、Radix UI、lucide-react、Python WebUI sidebar state JSON。

**依据文档：** `D:\project\worktrees\nanobot-study\Development Documentation\侧边栏新功能效果.md`

## Global Constraints

- 只做侧边栏、输入框项目选择、设置已归档对话相关能力。
- 不做独立后端 Project 实体。
- 不允许把已有会话随意改归属到另一个项目。
- 项目菜单里的“归档”是批量归档该项目下的未归档对话，不是归档项目容器。
- “移出侧边栏”只隐藏项目入口和该项目下未归档对话入口，不删除磁盘文件，不删除会话，不归档会话，不改会话归属。
- 已归档对话统一放到设置页管理，不再把“显示归档 / 隐藏归档”作为首页侧边栏主入口。
- 右键菜单、项目级“取消归档全部”、记录并恢复归档前置顶状态，都不进入第一期。
- Goose 和 Reasonix 只作为概念参考；实现以 nanobot 当前 WebUI 结构为准。

## 当前项目事实

- 首页侧边栏入口在 `webui/src/components/Sidebar.tsx`。
- 会话列表和项目分组主要在 `webui/src/components/ChatList.tsx` 与 `webui/src/lib/chat-groups.ts`。
- 侧边栏 UI 状态由 `webui/src/hooks/useSidebarState.ts` 管理，并通过 `nanobot/webui/sidebar_state.py` 持久化到 WebUI 状态 JSON。
- 当前已有会话级状态：`pinned_keys`、`archived_keys`、`title_overrides`、`project_name_overrides`、`collapsed_groups`。
- 当前 `WorkspaceControls.tsx` 已有空白页项目选择雏形，但还没有“搜索项目 / 不使用项目 / 添加空项目并显示在侧边栏”的完整交互。
- 当前 `SettingsView.tsx` 没有“已归档对话”分区，需要新增设置页 section，并从 `App.tsx` 传入会话和归档操作。
- 现有归档是前端侧边栏状态，不需要第一期新增后端会话归档 API。

## 设计边界

### 数据边界

- 会话真实归属仍以 `ChatSummary.workspaceScope.project_path` 为准。
- 默认工作区路径 `defaultWorkspacePath` 不当作项目展示；这些会话进入“对话”段。
- 项目 key 使用现有的路径规范化逻辑，和 `project_name_overrides` 保持一致。
- 空项目只存在于 WebUI 侧边栏状态里；没有首条消息前不创建会话。
- 被移出侧边栏的项目，如果后续重新添加同一文件夹，或者从设置里取消归档该项目下对话，就恢复显示。

### 显示边界

- 置顶只改变显示位置，不改变会话归属。
- 项目置顶后，整个项目块进入“置顶”段，项目下未归档对话跟随项目显示。
- 如果项目已经置顶，项目下单个置顶对话不再额外重复显示。
- 单个对话置顶后从原普通段移到“置顶”段，但仍保留原项目或纯对话归属。
- 已归档会话只在设置的“已归档对话”里出现。

### 操作边界

- 项目“归档”需要二次确认，确认后只把项目下未归档会话 key 批量加入 `archived_keys`。
- 项目“移出侧边栏”需要和“删除”区分文案；非空项目移出必须二次确认。
- 项目移出后，该项目下未归档会话不移动到“对话”段。
- 已归档对话行本身不作为打开入口，只提供“取消归档”和“删除”。
- 删除对话继续走现有删除会话逻辑。
- 添加项目只支持“使用现有文件夹”，不做输入项目名后自动创建默认文件夹。
- 桌面软件环境通过 `window.nanobotHost.pickFolder()` 打开系统文件夹选择器；普通浏览器环境不能获取本地目录绝对路径，只保留手动输入绝对路径的兜底方式。

## 状态模型

在现有 `SidebarStatePayload` 上追加项目级状态，保持旧状态文件可读：

```ts
export interface SidebarProjectEntryPayload {
  path: string;
  name?: string | null;
  created_at?: string | null;
  updated_at?: string | null;
}

export interface SidebarStatePayload {
  schema_version: number;
  pinned_keys: string[];
  archived_keys: string[];
  title_overrides: Record<string, string>;
  project_name_overrides: Record<string, string>;
  tags_by_key: Record<string, string[]>;
  collapsed_groups: Record<string, boolean>;
  view: SidebarViewStatePayload;
  pinned_project_keys: string[];
  removed_project_keys: string[];
  explicit_projects: Record<string, SidebarProjectEntryPayload>;
  updated_at?: string | null;
}
```

字段含义：

- `pinned_keys`：会话级置顶，继续沿用。
- `archived_keys`：会话级归档，继续沿用。
- `pinned_project_keys`：项目级置顶，key 为规范化项目路径。
- `removed_project_keys`：被移出侧边栏的项目 key。
- `explicit_projects`：用户显式添加过的项目，用于显示空项目。
- `project_name_overrides`：继续作为项目重命名的展示名来源。

schema 版本建议升级到 `2`，但 normalizer 必须接受缺少新字段的旧状态文件。

## Sidebar Model 接口

`webui/src/lib/sidebar-model.ts` 负责把散落状态合成 UI 可直接消费的数据：

```ts
export interface SidebarProjectGroup {
  key: string;
  path: string;
  label: string;
  sessions: ChatSummary[];
  archivedSessions: ChatSummary[];
  isPinned: boolean;
  isExplicit: boolean;
  isRemoved: boolean;
  updatedAt: number;
}

export interface SidebarProjectOption {
  key: string;
  path: string;
  label: string;
  shortPath?: string;
  hasUnarchivedSessions: boolean;
  hasArchivedSessions: boolean;
  isExplicit: boolean;
  isRemoved: boolean;
}

export interface ArchivedConversationGroup {
  key: string;
  label: string;
  projectPath: string | null;
  sessions: ChatSummary[];
}

export interface SidebarModel {
  pinnedProjects: SidebarProjectGroup[];
  pinnedChats: ChatSummary[];
  workspaceProjects: SidebarProjectGroup[];
  conversations: ChatSummary[];
  projectOptions: SidebarProjectOption[];
  archivedGroups: ArchivedConversationGroup[];
  archivedCount: number;
}
```

核心规则：

- `pinnedProjects` 包含置顶项目，项目下只放未归档会话。
- `pinnedChats` 包含单独置顶且没有被置顶项目包住的未归档会话。
- `workspaceProjects` 包含未置顶、未移出侧边栏、有未归档会话或显式添加过的项目。
- `conversations` 只包含无项目归属、未归档、未单独置顶的纯对话。
- `projectOptions` 包含当前项目、空项目、只有归档对话的项目；用于输入框搜索。
- `archivedGroups` 按原项目或纯对话分组，用于设置里的已归档对话页。

这样拆分的原因是：侧边栏模型和状态是基础，输入框项目选择和设置归档页都依赖它；先把模型做稳，后面的 UI 不需要反复改数据结构。

## 第一期开不做

- 不做右键菜单；`...` 菜单已经满足核心需求。
- 不做项目颜色、标签、复杂排序。
- 不做新建“无文件夹项目”。
- 不做项目级“取消归档全部”。
- 不做已归档对话直接打开查看。
- 不做归档前置顶状态的历史恢复。
- 不新增后端 Project 表或 Project API。

## 风险点

- 最大风险不是 UI，而是状态优先级：置顶、归档、移出侧边栏、空项目必须由同一个 `SidebarModel` 统一计算。
- `SettingsView.tsx` 已经很大，新增已归档页时建议拆组件，避免继续膨胀。
- 项目 key 必须统一使用规范化路径，否则重命名、置顶、移出、恢复会出现同一路径多份状态。
- 移出项目不是删除，所有弹窗和菜单文案必须克制，避免用户误解为删除磁盘文件。

## 最终验收标准

- 用户能清楚区分置顶、工作空间项目、纯对话、已归档对话。
- 用户能创建空项目，并在未创建对话前看到该项目。
- 用户能从空白页决定新对话是否归属某个项目。
- 用户能把项目或对话置顶，且不会改变真实归属。
- 用户能归档单条对话，也能批量归档某个项目下的对话。
- 用户能在设置里找回或删除已归档对话。
- 用户能移出不想看的项目入口，且不会删除文件、删除会话、归档会话或把项目会话改成纯对话。
