# 自定义指令功能方案

日期: 2026-07-06

## 结论

这个功能应该从“配置字段里的自由文本”改成“全局 `AGENTS.md` 文件 + 结构化思考语言选项”。

前端仍然保持用户确认过的形态:

- 设置页左侧增加“个性化”入口。
- 点击“个性化”后, 右侧显示“自定义指令”标题、输入框和保存按钮。
- 输入框编辑的是全局 `AGENTS.md` 文件内容。
- 额外提供一个“思考语言”选项按钮: `默认`、`中文思考`、`English thinking`。

后端不再把自定义指令保存为 `agents.defaults.custom_instructions` 这种普通 config 字段。实现上先抽象成 `GlobalInstructionsStore` 存储接口, 设置页和 prompt 拼接只依赖这个接口。第一版只提供本地文件实现, 全局自定义指令保存为 Nanobot 当前配置目录下的文件:

```text
get_config_path().parent / "AGENTS.md"
```

默认情况下就是:

```text
C:\Users\<user>\.nanobot\AGENTS.md
```

多实例或指定 config path 时, 本地文件版的全局 `AGENTS.md` 跟随该配置文件所在目录, 这样才符合“项目还没创建也能全局生效”的要求。以后如果发布网站版, 可以新增 DB/account 版 store, 不需要让前端或 `ContextBuilder` 继续绑定本机绝对路径。

## 关键事实

### 当前 prompt 拼接位置

系统提示词由 `nanobot/agent/context.py` 的 `ContextBuilder.build_system_prompt()` 拼接。

测试版 `custom_instructions` 配置字段不作为正式方案保留。当前实现已经改成:

- 全局指令内容通过 `GlobalInstructionsStore` 读取。
- 本地文件版 store 读取 config data dir 下的 `AGENTS.md`。
- 思考语言只保留结构化字段 `agents.defaults.reasoning_language`。

当前实际顺序是:

```text
Identity
→ Global Instructions: config data dir AGENTS.md
→ workspace bootstrap: AGENTS.md / SOUL.md / USER.md
→ Reasoning Language Preference  仅 zh/en 时出现
→ tool contract
→ memory/MEMORY.md
→ active skills
→ skills summary
→ recent history
→ session summary
```

注意: 当前用户消息不是 `build_system_prompt()` 的一部分, 而是在 `build_messages()` 里作为 user message 追加到 messages 后面。

### 现有项目级文件

`ContextBuilder.BOOTSTRAP_FILES` 当前是:

```python
["AGENTS.md", "SOUL.md", "USER.md"]
```

这些文件来自 workspace 根目录, 是项目级上下文, 不能承担“项目还没创建也生效”的全局入口。

现有几个文件应复用, 不需要改名、不需要改语义:

```text
workspace/AGENTS.md
workspace/SOUL.md
workspace/USER.md
workspace/memory/MEMORY.md
```

### memory/MEMORY.md 必须纳入链路

`nanobot/agent/memory.py` 的 `MemoryStore.__init__()` 明确使用:

```text
workspace / "memory" / "MEMORY.md"
```

当前 `MemoryStore.get_memory_context()` 会把它包装成:

```text
## Long-term Memory
...
```

然后 `ContextBuilder.build_system_prompt()` 再用:

```text
# Memory
...
```

注入 system prompt。

因此新方案的加载链路必须包含 `memory/MEMORY.md`, 不能只写全局 `AGENTS.md` 和项目级 `AGENTS/SOUL/USER`。

### 配置目录事实

`nanobot/config/loader.py` 的 `get_config_path()` 逻辑是:

```text
如果 set_config_path() 设置过路径: 使用该路径
否则: Path.home() / ".nanobot" / "config.json"
```

`nanobot/config/paths.py` 的 `get_data_dir()` 返回:

```text
get_config_path().parent
```

所以全局 `AGENTS.md` 应该绑定到 config data dir, 而不是 workspace。

## 参考 Codex 的实际行为

Codex 官方文档里的核心行为是:

- 全局 scope: Codex home 目录下的 `AGENTS.override.md` 优先于 `AGENTS.md`。
- 项目 scope: 从项目根目录走到当前工作目录, 每层最多取一个指令文件。
- 合并顺序: 从全局到项目根目录, 再到更近的目录。越靠后的内容越接近当前任务, 因此覆盖优先级更高。
- 空文件跳过。
- 有默认大小上限, Codex 默认 `project_doc_max_bytes = 32 KiB`。

Nanobot 第一版不建议一次性照搬完整目录遍历、override、fallback 文件名和大小上限。当前项目架构更适合先做最小稳定版本:

```text
全局 AGENTS.md
→ workspace/AGENTS.md
→ workspace/SOUL.md
→ workspace/USER.md
→ workspace/memory/MEMORY.md
→ 当前消息
```

这样能满足当前目标, 也不会改坏已有项目级 md 文件。

当前用户消息继续作为最后的 user message 进入 messages, 不塞进 system prompt。

## 优先级说明

需要在 prompt 里明确告诉模型不同来源的含义:

```text
Instruction priority inside this system prompt:
- Current user request is the immediate task context.
- Workspace instructions override global instructions for this workspace.
- Files listed later are more specific than files listed earlier.
- Memory is long-term context, not a higher-priority rule source.
- If memory conflicts with explicit global, workspace, or current user instructions, follow the explicit instructions.
```

中文 UI 里可以表达为:

```text
全局 AGENTS.md 是个人长期偏好。
workspace/AGENTS.md、SOUL.md、USER.md 是项目级上下文。
memory/MEMORY.md 是长期记忆上下文, 不应覆盖明确指令。
当前消息是本轮任务输入。
```

## 前后端绑定

### Settings payload

`settings_payload()` 建议新增:

```json
{
  "personalization": {
    "global_agents": {
      "path": "C:\\Users\\ai02\\.nanobot\\AGENTS.md",
      "content": "...",
      "exists": true
    },
    "reasoning_language": "default"
  }
}
```

也可以放在 `agent` 下, 但不建议继续叫 `custom_instructions`, 因为内容已经不是 config 字段, 而是全局指令文档内容。

`path` 第一版为本地文件版兼容字段, 实际来自 `GlobalInstructionsDocument.source`。如果后续做网站版, 建议把返回字段扩展为 `source` 或让 UI 文案从“保存位置”改成“保存范围”。

推荐命名:

```text
personalization.global_agents.content
personalization.global_agents.path
personalization.reasoning_language
```

### Update API

`update_agent_settings()` 支持:

```text
global_agents_content
reasoning_language
```

行为:

- `global_agents_content` 通过 `get_global_instructions_store().write(content)` 保存。
- 本地文件版 store 写入 `get_config_path().parent / "AGENTS.md"`。
- 本地文件版写入使用 UTF-8。
- 本地文件版保存前确保目录存在。
- 空内容可以写成空文件, 也可以删除文件。建议第一版写成空文件并在加载时按空内容跳过, 逻辑更简单。
- `reasoning_language` 写入 config。

### Config schema

删除或替换测试字段:

```python
custom_instructions: str = ""
```

新增结构化字段:

```python
reasoning_language: Literal["default", "zh", "en"] = "default"
```

含义:

- `default`: 不额外注入思考语言提示, 保持原始行为。
- `zh`: 注入“如果模型暴露或生成 reasoning/thinking 内容, 优先使用中文组织思考过程。不要因此改变最终回复语言。”
- `en`: 注入“如果模型暴露或生成 reasoning/thinking 内容, 优先使用 English for reasoning. Do not change the final answer language because of this setting.”

这个字段只控制思考语言偏好, 不控制回复语言。回复语言仍由用户当前消息、全局 `AGENTS.md`、项目文件等普通指令决定。

## 后端实现点

### 新增全局指令存储抽象

新增 `nanobot/config/global_instructions.py`, 把“全局指令从哪里读写”收敛到一个边界:

```python
@dataclass(frozen=True)
class GlobalInstructionsDocument:
    source: str
    content: str
    exists: bool


class GlobalInstructionsStore(Protocol):
    def read(self) -> GlobalInstructionsDocument: ...
    def write(self, content: str) -> bool: ...


class LocalFileGlobalInstructionsStore:
    def __init__(self, path: Path): ...
    def read(self) -> GlobalInstructionsDocument: ...
    def write(self, content: str) -> bool: ...


def get_global_instructions_store() -> GlobalInstructionsStore:
    return LocalFileGlobalInstructionsStore(get_global_agents_path())
```

`nanobot/config/paths.py` 保留本地文件路径 helper:

```python
def get_global_agents_path() -> Path:
    return get_data_dir() / "AGENTS.md"
```

这个 helper 只属于本地文件版实现。以后网站版可以新增例如 `DatabaseGlobalInstructionsStore(account_id)` 或 `RemoteGlobalInstructionsStore(...)`, 但 `settings_api`、`AgentLoop.from_config()`、`ContextBuilder` 不应该再直接读写文件路径。

如果后续要更接近 Codex, 可以再扩展:

```text
AGENTS.override.md
AGENTS.md
```

第一版建议先只做本地文件版 `AGENTS.md`。

### ContextBuilder 改造

`ContextBuilder.__init__()` 接收:

```python
global_instructions_store: GlobalInstructionsStore | None = None
global_agents_path: Path | None = None
reasoning_language: Literal["default", "zh", "en"] = "default"
```

`global_agents_path` 只作为本地文件兼容入口。如果传入 `global_instructions_store`, 优先使用 store。

`AgentLoop.from_config()` 从 config 传入:

```python
global_instructions_store=get_global_instructions_store()
reasoning_language=defaults.reasoning_language
```

`build_system_prompt()` 中:

1. 通过 `GlobalInstructionsStore.read()` 读取全局指令。
2. 读取 workspace `AGENTS.md/SOUL.md/USER.md`。
3. 注入思考语言偏好。
4. 保留 tool contract、memory、skills、recent history 的原有位置。

### bootstrap loader 改造

当前 `_load_bootstrap_files(root)` 只返回一段字符串, 且只标 `## {filename}`。

建议改为 source-aware:

```text
_load_global_agents(store) -> str
_load_workspace_bootstrap_files(root) -> str
_format_instruction_section(title, source, content) -> str
```

输出示例:

```md
# Global Instructions
Source: C:\Users\ai02\.nanobot\AGENTS.md

...
```

`Source` 来自 `GlobalInstructionsDocument.source`。本地文件版显示 config dir 下的 `AGENTS.md`; 网站/账号版以后可以显示账号级来源, 例如 `account:global`。这样模型能知道哪些是全局, 哪些是项目级别。

### 当前实现落点

已落地的源码边界:

```text
nanobot/config/global_instructions.py
  - GlobalInstructionsDocument
  - GlobalInstructionsStore
  - LocalFileGlobalInstructionsStore
  - get_global_instructions_store()

nanobot/config/paths.py
  - get_global_agents_path()

nanobot/webui/settings_api.py
  - _global_agents_payload()
  - _write_global_agents_content()
  - settings_payload().personalization.global_agents
  - update_agent_settings().global_agents_content

nanobot/agent/loop.py
  - AgentLoop.from_config(..., global_instructions_store=...)

nanobot/agent/context.py
  - ContextBuilder(..., global_instructions_store=...)
  - _load_global_agents()
```

设计边界:

- 设置 API 只调用 `get_global_instructions_store().read()/write()`。
- `AgentLoop.from_config()` 默认使用本地文件 store, 但允许调用方注入自定义 `global_instructions_store`。
- `ContextBuilder` 只依赖 store 读取结果, 不关心内容来自文件、数据库还是远程账号配置。
- `global_agents_path` 只保留为本地文件兼容入口, 不是未来网站版的主接口。

## 与 Codex 的差异

第一版实现后, 效果类似 Codex 的核心体验, 但不是完整复制。

相同点:

- 都有一个全局 `AGENTS.md`。
- 都把全局指令放在项目级指令之前。
- 都通过 prompt 拼接生效。
- 都适合写长期偏好, 例如 KISS、回复风格、确认策略、常用验证习惯。

差异:

- Codex 全局目录是 `~/.codex`, Nanobot 建议是当前配置目录 `~/.nanobot` 或指定 config path 的父目录。
- Codex 支持 `AGENTS.override.md`, 第一版 Nanobot 暂不做。
- Codex 支持从项目根目录走到当前工作目录的目录级链, 第一版 Nanobot 只加载 workspace 根目录文件。
- Codex 支持 fallback 文件名和大小上限, 第一版 Nanobot 暂不做。
- Codex memories 是 Codex home 下的生成状态; Nanobot 当前使用 workspace 内的 `memory/MEMORY.md`, 并且已经注入 system prompt。

## 效果边界

### 可以实现的效果

- 设置页保存全局指令后, 后续新 agent turn 都能读到。
- 本地文件版项目还没创建时, 全局 `AGENTS.md` 仍然能保存, 因为它绑定 config dir, 不绑定 workspace。
- workspace 创建后, 项目级 `AGENTS.md/SOUL.md/USER.md/memory/MEMORY.md` 会继续叠加。
- 模型会在 prompt 里看到每段来源和优先级。
- `默认 / 中文思考 / English thinking` 可以作为稳定的思考语言偏好注入。

### 不能保证的效果

- 不能保证模型真实隐藏思考链的内部语言。很多 provider 不暴露真实 chain-of-thought, 有些只暴露 summary 或 reasoning delta。
- 不能保证所有模型都严格遵守思考语言提示。它是 prompt 级偏好, 不是通用 provider API 参数。
- 不能保证旧会话立刻重载。当前架构下, 最稳妥的规则是: 保存后对新 turn 生效; 如果 runtime 没有热更新 ContextBuilder, 则需要重启 gateway 或重建 AgentLoop。

## 后续可优化

如果第一版稳定, 再考虑补齐 Codex 更完整的行为:

- 支持 `AGENTS.override.md`。
- 支持 workspace 子目录级 `AGENTS.md` 链。
- 支持 fallback 文件名配置。
- 支持 `project_doc_max_bytes` 上限。
- 设置页显示“当前生效的指令来源列表”。
- 在调试模式下输出当前 prompt 使用了哪些文件, 方便排查“为什么这条规则生效/没生效”。

这些都不是第一版必须项。当前最重要的是先把全局入口、项目入口、memory 入口和思考语言选项的边界做清楚。
